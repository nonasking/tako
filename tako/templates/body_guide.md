# tako 본문 작성 가이드 (개인 커스텀 가능)

이 파일은 `/tako`(생성)·`/tako-update`(수정) 가 *제목·본문을 만들 때 따르는 규칙*이다.
세션 컨텍스트를 티켓으로 옮길 때 LLM 이 이 가이드를 읽고 그대로 적용한다.

- 이 파일은 **개인용**이다. 자유롭게 고쳐서 본인 팀 스타일로 맞춰도 된다.
- 편집: 에디터로 직접 고치거나, Claude Code 에서 `/tako-guide` 로 대화하며 고친다.
- 기본값으로 되돌리려면 `tako guide reset`.
- 경로는 `tako guide path` 로 확인.

아래 네 섹션(제목 / 본문 구조 / 본문 작성 규칙 / 자가 점검)이 본문 생성의 전부다.
섹션을 지우거나 규칙을 바꾸면 그 변경이 곧바로 티켓 초안에 반영된다.

---

## 제목

작업 *개요* 가 한 줄로 드러나게.

- 형태: *대상 + 동사(또는 결과)*. 명사구 60자 이내, 마침표 없음.
- 모호 단어 단독 금지: "기능 개선", "버그 수정", "리팩터링", "코드 정리".
- 좋은 예
  - 보드 정렬 토큰 격리로 다른 컬럼 영향 제거
  - WL-9200 자식 이슈 일괄 조회 JQL 추가
  - 결제 환불 처리 가상계좌 케이스 누락 보완
- 나쁜 예
  - 버그 수정
  - 정렬 기능 개선
  - 코드 정리 및 리팩터링

## 본문 구조

본문 최상단에 아래 두 섹션을 *반드시* 둔다. 다른 섹션은 필요할 때만 그 아래에 추가(옵션).

```
## 내가 한 일
- <세션에서 실제로 한 작업을 *무엇을·왜* 중심으로 한 줄. 코드·파일명 나열 대신 평이한 말로>
- <검증 완료된 사실만. 가설·추정 금지>

## 결론
- <지금 상태: 완료 / 진행 중 / 막힘 / 후속 필요>
- <남은 작업·다음 액션이 있으면 한 줄>

(옵션) ## 배경
(옵션) ## 완료 조건
(옵션) ## 영향 범위 — 단, 세션에서 *직접 확인한* 영향만
```

- 두 섹션은 *반드시* `## ` 마크다운 헤더로 쓴다. `[내가 한 일]` 같은 *대괄호 라벨* 로 만들지 말 것 — 대괄호는 Jira ADF 가 헤더로 렌더링하지 못하고 일반 문단으로 박힌다.
- 두 섹션이 *비어 있는* 본문 금지. 둘 다 *한 줄 이상* 채울 내용이 없으면 사용자에게 본문을 한 번 묻는다.

## 본문 작성 규칙

티켓은 *개발자가 아닌 사람(기획자 등)도 읽고 이해할 수 있어야 하는* 문서다.
AI 가 만든 문장이라도 *사람이 직접 쓴 것처럼 자연스럽게* 읽혀야 하고, 검증되지 않은 내용으로 본문을 부풀리지 않는다.

- 세션 안에서 *실제로 실행했거나 코드에서 확인한 사실* 만 쓴다. 실행 안 한 가설·일반론·추측은 빼거나 `> 미검증:` 같은 별도 라벨로 분리.
- 본 작업과 *직접 관련 없는 주변 분석* 은 본문에 박지 않는다 (예: 작업한 함수와 무관한 코드 품질 평가, 보지도 않은 모듈 추측).
- "~일 것 같다", "추정", "아마도", "may", "could" 같은 모호 표현은 *처음부터 쓰지 않는다*. 필요해 보이면 사용자에게 한 번 묻고 *검증된 사실로* 다시 쓴다.
- 본문 길이는 *실제 작업 분량에 비례*. 작은 변경에 일반론 길게 늘이지 않는다. 한 줄로 충분하면 한 줄.
- **깔끔한 마크다운으로.** 불릿·짧은 문단 위주로 읽기 좋게. (최상단 두 섹션은 `## 내가 한 일` / `## 결론` 마크다운 헤더로 — 위 형식 그대로.)
- **코드·로그를 그대로 붙여넣지 않는다.** AI 코딩 도구 출력이든 직접 친 코드든 raw 코드·스택트레이스 통째 복붙 금지. 꼭 필요한 식별자(파일·함수명) 정도만 남기고 *무엇을·왜* 했는지 평이한 문장으로 푼다.
- **개발자가 아닌 기획자도 이해할 언어로.** 기술 디테일은 최소화하고 *의미*(무슨 문제를 어떻게 풀었는지) 중심으로 쓴다. AI 가 찍어낸 듯한 번역체·정형구 말고 *사람이 직접 쓴 것처럼* 자연스럽게. 단, 자연스럽게 쓴다고 위 분량 비례 원칙을 어기며 길게 늘이지 않는다.

## 자가 점검 체크리스트

미리보기를 띄우기 *전에* 작성한 제목·본문을 한 번 훑는다. 아래가 *하나라도* 걸리면 미리보기에 *짧은 경고 라인* 을 같이 띄워 사용자가 결정하게 한다 — 자동으로 고치지 말 것.

1. **제목이 작업 개요를 보여주는가?** 모호 단어("개선", "수정", "리팩터링", "정리")만 있고 *대상·결과* 가 없으면 경고.
2. **제목이 너무 짧은가?** 10자 미만이면 경고 — 대개 정보 부족.
3. **본문 최상단에 필수 두 섹션이 있는가?** `## 내가 한 일` 과 `## 결론` 이 *본문 시작 부분* 에 둘 다 있어야 함. 하나라도 누락 / 내용 한 줄 미만이면 경고. 또한 섹션이 `[내가 한 일]` 같은 *대괄호 라벨* 로 돼 있으면 `## ` 헤더로 바꿔 진행할지 경고.
4. **본문에 미검증 표현이 있는가?** "~일 것 같다", "추정", "아마도", "may", "could" 등. 발견 시 *어느 줄인지* 같이 알려주기.
5. **본문이 세션 작업 분량 대비 과한가?** 한 함수 수정인데 본문에 모듈 전반 분석이 들어가 있으면 경고.
6. **본 작업과 무관한 분석이 섞였는가?** 사용자가 *명시적으로 요청* 하지 않은 주변 코드 평가·미실행 가설은 빼거나 한 번 묻기.
7. **코드 복붙·전문용어 과다인가?** raw 코드/로그가 통째로 들어갔거나, 기획자가 못 알아볼 전문용어 위주면 경고 — 평이한 언어로 풀지 묻기.

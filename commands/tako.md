# /tako — 세션 컨텍스트 기반 Jira 티켓 생성

이 커맨드의 가치는 한 가지: **현재 세션 작업을 LLM 이 요약해 티켓 초안을 만든다**. 그 외(인증, 페이로드 조립, REST 호출, ADF 변환) 는 `tako` 셸 명령이 처리. 본 마크다운은 셸 호출 전 단계인 *제목·본문 후보 생성* 과 *사용자 확인* 만 다룬다.

## 작동 환경 가정

- `tako init` 으로 설정·인증 파일이 이미 만들어져 있는 상태
- 사용자 본인 권한으로 Atlassian Cloud REST API 호출이 가능한 상태
- 대상 프로젝트는 team-managed 가정 (일반 이슈가 Epic 을 부모로 가지면 `parent` 필드 하나로 표현)

## 흐름

### 1. 입력 수집 — 두 갈래

#### 1.A 세션에 작업 맥락이 *있을 때* (이 커맨드의 본 가치)

- 세션 맥락을 짧게 요약해 **제목 후보**를 만든다 (한국어, 60자 이내, 마침표 없는 명사구).
- **본문 후보**는 마크다운으로 작성한다. 일반적으로 다음 구조를 쓰되 필요 없는 섹션은 생략한다.
  ```markdown
  ## 배경
  ...
  ## 작업 내용
  ...
  ## 완료 조건
  - [ ]
  ```
- 사용자가 자유 인자로 명시한 정보(예: "CATS 프로젝트", "이거 부모는 infra")를 우선 적용한다.
- 빠진 항목(프로젝트, 이슈 유형, 부모)은 사용자 설정의 기본값으로 채운다.
- **본문이 비어 있는 후보를 만들지 말 것.** 세션에서 추출할 내용이 정말 없으면 사용자에게 본문을 한 번 묻는다.

#### 1.B 세션 맥락이 *없을 때*

본 슬래시 커맨드가 가져다 주는 가치가 거의 없으므로 사용자에게 다음을 안내한다.

> 세션 컨텍스트가 비어 있어서 `/tako` 가 특별히 더 해 드릴 게 없습니다. 셸에서 다음 한 줄이 더 가볍습니다:
>
> ```bash
> tako new
> ```

사용자가 그래도 진행을 원하면 1.A 와 같은 인터랙티브 입력으로 진행한다.

### 1.5 선택 항목 한 번 묻기 — 미리보기 *직전*

필수 항목(프로젝트·유형·제목·본문) 이 정해진 *직후*, **미리보기를 보여주기 전에** 사용자에게 *선택 항목*을 한 번 모아서 묻는다. 사용자가 *명시적으로 발화*했으면 묻지 않고 그 값을 그대로 사용. 비어 있으면 미리보기에 안 들어가고 페이로드에서 빠짐.

묻는 형식 (한국어, 한 메시지에 묶기):

```
선택 항목을 추가하시겠어요? 안 쓰면 비워두세요.
  - 담당자 (me / 이메일 / accountId, 예: me 또는 jy@example.com):
  - 스토리포인트 (정수, 예: 3):
  - 기한 (YYYY-MM-DD, 예: 2026-06-15):
  - 연결할 티켓 (KEY[:TYPE] 쉼표로 여러 개, 예: WL-200, WL-300:Blocks):
```

원칙:
- 사용자가 *세션 맥락 또는 인자로 이미 말한 값*은 *다시 묻지 않는다* (예: "SP 3" 이라고 했으면 SP 입력 칸 생략).
- 묻기는 *한 번*만. 사용자가 "그냥 만들어" / "비워둬" / 빈 응답으로 답하면 *전부 스킵*하고 미리보기로 진행.
- 사용자 답변이 모호하면 다시 묻지 말고 *해당 항목만 비움*. 미리보기 단계에서 다시 추가 가능.
- 세션 맥락에서 *추정해서 자동으로 채우지 말 것* (예: 세션에 "내일까지" 라는 표현이 있어도 사용자가 직접 말하지 않은 한 기한으로 박지 않는다).

### 2. 미리보기

수집한 정보를 한국어 미리보기로 사용자에게 보여준다. 실제 미리보기 렌더링은 `tako preview` 가 책임진다. heredoc + `'TAKO_INPUT'` 으로 따옴표/줄바꿈을 안전하게 전달한다.

```bash
cat <<'TAKO_INPUT' | tako preview
{
  "project": "<프로젝트 키>",
  "issue_type": "<이슈 유형>",
  "summary": "<제목>",
  "description": "<마크다운 본문>",
  "parent_epic": "<별칭 또는 이슈 키, 없으면 생략>",
  "labels": ["<선택>"]
}
TAKO_INPUT
```

stdout 에 나온 미리보기를 그대로 사용자에게 노출한다.

### 3. 사용자 확인 (Y/n)

미리보기 직후 한국어로 묻는다.

```
이대로 Jira에 만들까요? (Y/n)
   y: 그대로 생성 → 4번 단계
   n: 취소
   수정 의향 있는 답변(예: "제목을 ~로 바꿔", "부모 빼고 다시"): 입력 보정 후 1번 미리보기 재생성
```

`y` 또는 빈 응답(Enter)일 때만 다음 단계로. 그 외에는 흐름을 멈추거나 보정 후 재생성한다.

### 4. 실제 생성 — `tako new` 호출

확인이 떨어지면 같은 정보를 인자로 묶어 `tako new` 를 호출한다. heredoc 으로 description 만 stdin 처럼 넘기지 못하므로 인자로 전달하되, **반드시 single-quote heredoc 래퍼 또는 작은따옴표 escape 패턴으로 줄바꿈/특수문자가 깨지지 않도록 한다**.

가장 안전한 패턴:

```bash
tako new \
  --project '<프로젝트 키>' \
  --issue-type '<이슈 유형>' \
  --summary "$(cat <<'SUM'
<제목>
SUM
)" \
  --description "$(cat <<'DESC'
<마크다운 본문>
DESC
)" \
  --parent '<부모 키 또는 별칭>' \
  --assignee '<me|이메일|accountId, 사용자가 명시했을 때만>' \
  --story-points <정수, 사용자가 명시했을 때만> \
  --duedate <YYYY-MM-DD, 사용자가 명시했을 때만> \
  --link '<KEY[:TYPE], 사용자가 명시했을 때만, 반복 가능>' \
  --yes
```

- `--assignee` / `--story-points` / `--duedate` / `--link` 는 *1.5 단계에서 사용자가 입력한 값* 또는 *사용자가 발화로 명시한 값* 일 때만 인자에 포함한다. 세션 맥락에서 *추정해서* 채우지 말 것.
- `--assignee` 의 한국어 이름은 v1.x 미지원. 사용자가 "민성한테", "주열이가" 같은 한국어 이름으로 말하면 *이메일 또는 accountId 를 받아오라*고 사용자에게 한 번 묻는다.
- `--link` 의 TYPE 은 생략 시 `Relates`. 사용자가 "blocks", "duplicates" 같은 단어를 쓰면 적절한 영문 TYPE 으로 변환 (`Blocks` / `Duplicates` 등).
- `--yes` 를 붙이는 이유: 미리보기·확인 단계는 *이 슬래시 커맨드가 이미 처리*했으므로 `tako new` 의 자체 확인 단계는 건너뛴다. 두 번 묻지 않는다.
- description 의 마크다운→ADF 변환은 `tako new` 가 내부적으로 `md-to-adf` 로 처리한다. LLM 이 별도로 변환할 필요는 없다.

### 5. 결과 보고

`tako new` 가 stderr 에 `생성 완료 / 키 / 링크` 를 한국어로 쓰고 stdout 에는 이슈 키만 출력한다. 사용자에게 정리해서 노출:

```
생성 완료
  키:   <KEY>
  링크: https://<site>/browse/<KEY>
```

## 의사결정 사전 상의 (반드시 지킬 것)

LLM(이 마크다운을 따르는 너 자신)이 *추정으로* 진행하지 말고 사용자에게 먼저 묻고 가야 하는 상황:

- **부모 에픽이 모호** — alias 미존재 / 여러 후보 / 입력값이 키 패턴과 다름.
- **이슈 유형이 사용자 설정에 정의되지 않음** — 미리 안내하고 사용자가 설정에 추가하도록 권장.
- **세션 맥락이 모호한 채 자동 채우기** — 작업 맥락이 너무 광범위하거나 여러 갈래일 때, 어떤 흐름을 티켓으로 만들 건지 한 번 확인.
- **본문 후보가 비어 있을 때** — 4번 단계로 진행하지 말 것. 본문을 사용자에게 묻는다.
- **사용자가 "데모 / 시뮬레이션 / 흐름만"** 같은 표현을 써도 임시 설정·임시 인증으로 우회하지 않는다. 설정·인증이 없으면 안내만 하고 종료한다.

## 후속 요청 처리

사용자가 "다시 만들어줘", "방금 거 부모만 바꿔서", "이번엔 Bug 로" 같은 후속 표현을 쓰면 이전 입력을 메모리에 가지고 있다가 변경된 항목만 덮어써서 1번부터 다시 돌린다. 백지에서 다시 묻지 않는다.

## 실패 처리

- `tako preview` / `tako new` 가 비-제로 종료 코드를 반환하면 stderr 를 그대로 사용자에게 보여주고 흐름을 중단한다.
- `tako new` 가 422/400 을 받으면 응답 본문을 한국어로 요약해 사용자에게 보여주고, 재시도 여부를 묻는다. 자동 재시도는 하지 않는다 (실수로 같은 티켓이 두 번 만들어지는 사고 방지).
- 인증 실패(401/403) 면 `tako init` 으로 자격 증명을 다시 만들도록 안내한다. `tako` 명령이 *PATH 에 없는* 환경이면 `python -m tako ...` 형태로 폴백 호출도 안내한다.

# /tako-retype — 기존 티켓의 유형 변경

이미 만들어진 티켓의 *유형*만 바꾼다. 제목·본문은 건드리지 않는다. 유형 변경은 새 유형의 워크플로·필수 필드·화면 설정에 따라 거부될 수 있고 한 번 바꾸면 되돌리려면 다시 바꿔야 하므로, 미리보기 → Y/n 을 강제한다. *어떤 유형으로 바꿀 수 있는지*(허용 목록)와 경계 거부 판정은 셸 명령 `tako retype` 가 책임진다 — 본 커맨드는 사용자 의도 파악과 확인 흐름만 맡는다.

## 작동 환경 가정

- `tako init` 완료. 본인 사이트 인증 가능 상태
- 세션 컨텍스트는 *필요 없다* — 키와 바꿀 유형만 있으면 동작. 셸에서 `tako retype WL-1234 Story` 직접 호출이 더 가벼우면 그쪽을 권유한다.

## 흐름

### 1. 인자 정규화

사용자 입력에서 *이슈 키*(WL-8876 / browse URL 둘 다 OK)와 *바꿀 유형 이름*을 뽑는다. 둘 중 하나라도 불분명하면 추정하지 말고 한 번 되묻는다 ("어느 티켓을, 무슨 유형으로 바꿀까요?").

### 2. 현재 상태 미리보기

`tako show <KEY> --json` 으로 현재 유형·제목을 받아 사용자에게 보여준다.

```
[WL-1234] 정렬 로직 리팩터링
  현재 유형: Task → 바꿀 유형: Story
```

- 현재 유형과 바꿀 유형이 같으면 그 사실만 알리고 종료 (바꿀 게 없음).
- 이 단계에서는 *허용 여부를 단정하지 않는다*. 실제 허용 목록 검증은 다음 단계의 `tako retype` 가 한다.

### 3. 사용자 확인 (Y/n)

위 미리보기를 보여주고 "이대로 유형을 바꿀까요?" 로 Y/n 을 받는다. 되돌리려면 다시 바꿔야 한다는 점, 새 유형 설정에 따라 거부될 수 있다는 점을 한 줄로 덧붙인다.

### 4. 적용 — `tako retype` 호출

확인받으면 `--yes` 로 셸 자체 확인은 건너뛰고 호출한다 (슬래시가 이미 받았으므로 중복 안 함). `--yes` 여도 셸 명령이 editmeta 로 *허용 여부를 다시 검증*하므로 막힌 변환을 강제하지는 않는다.

```bash
tako retype <KEY> "<새 유형>" --yes
```

### 5. 결과 보고

성공:
```
유형 변경 완료: <KEY>
  Task → Story
  링크: https://<site>/browse/<KEY>
```

## 실패 처리

`tako retype` 가 비-제로로 끝나면 stderr 를 *그대로* 사용자에게 전달하고 멈춘다. 특히 다음 경우는 자연어로 한 번 더 풀어 설명한다:

- **부분 반영(경고)** — PUT 은 통과했지만 사후 재조회에서 유형이 요청과 다른 경우(워크플로/화면 설정이 일부만 반영). 이때 셸은 stdout 에 키를 출력하되 *비-제로로 끝나 성공이 아님*을 표시한다. "변경이 일부만 반영됐습니다 — Jira 에서 직접 확인하세요" 로 안내하고 링크를 보여준다.

- **바꿀 수 없는 유형** — 셸이 "바꿀 수 있는 유형: A, B, C" 목록을 준다. 그 목록을 보여주고 그중에서 다시 고를지 묻는다. 추정으로 다른 유형을 임의 선택하지 않는다.
- **계층 경계 변환** — 하위작업↔표준 처럼 계층을 넘는 변환은 목록에 없으며 Jira UI 의 *이동(Move)* 이 필요하다고 안내한다. tako 는 이 변환을 하지 않는다.
- **워크플로/필수 필드 거부 (400/422)** — 새 유형의 워크플로에 현재 상태가 없거나 필수 필드가 비어 거부된 경우. Jira 메시지를 그대로 노출하고, 필요하면 해당 필드를 먼저 채운 뒤 다시 시도하도록 안내한다.

## 의사결정 사전 상의 (반드시)

- **바꿀 유형이 불분명** — "유형 바꿔줘" 정도로만 말하면 현재 유형을 보여주고 어떤 유형으로 바꿀지 받는다.
- **허용 목록에 없음** — 셸이 준 허용 목록만 제시하고 그 안에서 고르게 한다. 비슷해 보이는 다른 유형을 대신 고르지 않는다.
- **여러 티켓 의도** — 한 번에 여러 키를 말하면 한 건씩 확인하며 진행할지 먼저 묻는다.

## 후속 요청

- "다른 유형으로 다시" — 같은 키에 새 유형만 바꿔 4번부터 다시
- "다른 티켓도 같은 유형으로" — 키만 바꿔 같은 흐름 재실행
- "제목/본문도 같이" — 유형 변경 후 `/tako-update` 로 이어서 안내

## 범위 밖

- 제목·본문 변경 → `/tako-update`
- 새 티켓 생성 → `/tako`
- 계층을 바꾸는 변환(하위작업↔표준) → Jira UI 이동(Move). v1 미지원.

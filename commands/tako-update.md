# /tako-update — 세션 맥락 → 티켓 제목/본문 반영

세션에서 한 작업을 *티켓 본문(description)* 또는 *제목(summary)* 에 기록한다. 부수 가치 아닌 *명세 누적*이라는 한 가지에 집중. 코멘트 추가가 아니라 *티켓 자체*를 건드리는 점에서 위험이 있으므로 미리보기 → Y/n 강제.

## 작동 환경 가정

- `tako init` 완료. 본인 사이트 인증 가능 상태
- 세션 컨텍스트가 *있어야* 의미 있음. 빈 세션이면 사용자에게 안내하고 종료
- 대상 프로젝트는 team-managed 가정

## 흐름

### 0. 본문 작성 가이드 로드 — 본문 정리 *전*

제목·본문을 만들기 전에 사용자 개인 가이드를 읽어 *그대로 따른다*.

```bash
tako guide show
```

- stdout 의 가이드(제목 형태·본문 구조·작성 규칙·자가 점검)가 이 커맨드의 **작성 규칙 전체**다. 아래 흐름은 그 가이드를 *적용하는 절차* 만 다룬다 — 작성 규칙을 여기 따로 적지 않는다.
- stderr 에 출처(개인 / 기본값)가 나온다. 규칙 자체를 바꾸고 싶어 하면 `/tako-guide` 를 안내한다.
- 가이드를 못 읽는 드문 경우엔 tako 기본 원칙(필수 두 섹션 · 검증된 사실만 · 기획자도 이해할 평이한 본문)으로 진행한다.

### 1. 인자 정규화

사용자 인자에서 키 추출 (WL-8876 / browse URL 둘 다 OK). 형태 안 맞으면 사용자에게 한 번 확인.

### 2. 현재 본문 조회 + 세션 맥락 요약

- `tako show <KEY> --json` 으로 현재 본문 받음 (모드 분기·미리보기에 활용)
- 동시에 세션 맥락에서 *진행 상황·결정 사항* 등을 한국어 마크다운으로 정리. 구조·톤·금지사항은 *모두 0번에서 읽은 가이드를 따른다*. 가이드가 요구하는 필수 섹션(기본값 기준 마크다운 헤더 `## 내가 한 일` / `## 결론`)을 최상단에 둔다. 섹션 라벨을 `[내가 한 일]` 같은 *대괄호 표기* 로 만들지 말고 *반드시 `## ` 헤더* 로 작성한다 (Jira ADF 가 헤더로 렌더링).
  - **배치만 update 특유** — append 모드는 `## 업데이트 (YYYY-MM-DD)` *섹션 안* 에 가이드의 필수 섹션을 둔다. overwrite 모드는 *통째 새 본문* 의 최상단에 둔다.
- 세션이 비었거나 너무 짧으면 *진행하지 말고* 사용자에게 묻는다.

### 2.1 작성 원칙

제목·본문의 작성 원칙(제목 형태, 필수 섹션, 검증된 사실만, 미검증 표현 금지, 분량 비례, 코드 복붙 금지, 기획자도 이해할 평이한 언어 등)은 *0번에서 읽은 가이드에 전부 들어 있다*. 그 가이드를 그대로 따른다. 규칙을 바꾸고 싶으면 본문이 아니라 `/tako-guide` 로 가이드를 고친다.

*update 특유* 인 점만 추가로 유의: 제목 점검은 `--summary` 를 바꿀 때만, 본문 점검은 `--body` 를 바꿀 때만 적용한다 (사용자가 손대라고 한 쪽만).

### 3. 모드 결정

- 기본 **append** — 기존 본문 끝에 `## 업데이트 (YYYY-MM-DD)` 섹션 추가
- 사용자가 "통째 다시 써", "명세 재작성" 같은 표현 시에만 **overwrite** 분기 (`--mode overwrite`)
- 섹션 이름을 사용자가 지정하면 (`--section "진행 상황"` 등) 그대로 사용
- 사용자가 "제목도 바꿔줘", "제목만 ~로", "summary 를 ~로" 같이 *제목 변경* 의도 → `--summary "<새 제목>"` 인자 같이 전달. 본문 손대지 말라고 했으면 `--summary` 만, 같이 바꾸라고 했으면 `--summary` + `--body`

### 3.5 자가 점검 — 미리보기 *직전*

미리보기 띄우기 *전에* 새 제목·새 섹션·새 본문을 너 자신이 한 번 훑는다 (출력 없이). 다음이 *하나라도* 걸리면 미리보기에 *짧은 경고 라인* 을 같이 띄워 사용자가 결정하게 한다 — 자동 수정 금지.

체크리스트:

- **기본 점검** — 0번에서 읽은 가이드의 「자가 점검 체크리스트」를 그대로 적용한다 (제목·필수 섹션·미검증 표현·분량·코드 복붙 등). 단 제목 점검은 `--summary` 변경 시, 본문 점검은 `--body` 변경 시에만.
- **update 특유 점검** — 가이드 항목에 더해 한 가지를 본다: *append 새 섹션이 기존 본문과 중복되는 일반론인가?* 이미 본문에 적힌 배경 설명을 반복해 늘이지 말 것.

경고 출력 형태 (미리보기와 같이):

```
[자가 점검 — 사용자 결정 부탁]
  - 새 섹션 최상단에 `## 내가 한 일` / `## 결론` 두 섹션이 누락됐습니다. 추가하고 진행할까요?
  - 새 본문에 `[내가 한 일]` 같은 대괄호 라벨이 보입니다. `## 내가 한 일` 헤더로 바꿔 진행할까요?
  - 본문 "## 영향 평가" 부분은 세션에서 검증한 내용이 아니라 일반 분석입니다. 빼고 진행할까요?
  - 제목 "정렬 개선" 은 작업 결과가 잘 안 보입니다. 그대로 진행할지 알려주세요.
  - 새 섹션에 코드 블록이 통째로 들어가 있습니다. 기획자도 이해할 평이한 설명으로 바꿔 진행할까요?
```

원칙:
- *경고만*. 자동 수정·자동 삭제 금지.
- 걸리는 게 없으면 조용히 미리보기로 진행.

### 4. 미리보기 + 확인

`tako update <KEY> --body "..." --section "..."` 호출 *없이* 먼저 사용자에게 *추가될 섹션* 또는 *교체될 본문* 을 한국어로 보여주고 Y/n. 사용자가 동의하면 다음 단계로.

### 5. 적용 — `tako update` 호출

확인 받으면 인자 묶어서 호출. `--yes` 로 셸 자체 확인은 건너뛴다 (슬래시 커맨드가 이미 받았으므로 두 번 묻지 않음).

```bash
tako update <KEY> \
  --section "<섹션 이름>" \
  --mode append \
  --body "$(cat <<'BODY'
- 작업 내용 1
- 작업 내용 2
BODY
)" \
  --yes
```

overwrite 모드면 `--mode overwrite` + `--body` 에 *완성된 새 본문* 전체.

제목만 변경:
```bash
tako update <KEY> --summary "<새 제목>" --yes
```

제목 + 본문 동시 변경:
```bash
tako update <KEY> \
  --summary "<새 제목>" \
  --body "..." \
  --mode overwrite \
  --yes
```

`--summary` 와 `--body` 중 *최소 하나는* 있어야 함. `--mode` 는 본문에만 영향 — 제목은 항상 교체.

### 6. 결과 보고

```
업데이트 완료: <KEY>
  링크: https://<site>/browse/<KEY>
```

## 의사결정 사전 상의 (반드시)

다음 상황에서 LLM 이 추정하지 말고 사용자에게 묻는다:

- **세션 컨텍스트가 비었거나 매우 짧음** — "업데이트로 기록할 작업 흐름이 세션에 보이지 않습니다. 직접 본문을 알려주시거나 더 작업하신 뒤 다시 호출해 주세요."
- **세션이 여러 갈래** — 한 세션에 *여러 작업 흐름*이 섞여 있을 때. 어느 흐름을 본문에 반영할지 한 번 확인.
- **overwrite 모드 의도 불분명** — 사용자가 "업데이트해줘" 같은 모호한 표현. 기본은 append, 사용자가 *명시적으로 통째 재작성*을 원하는지 한 번 확인.
- **본문 현재 상태가 *명세성 정보로 가득* 한 경우** — overwrite 면 명세 손실. 사용자가 의도한 게 맞는지 추가 확인.
- **제목 변경 의도가 모호** — 사용자가 "제목 좀 바꿔" 정도로만 말하면 *현재 제목 그대로 보여주고* 새 제목을 받는다. 세션 맥락에서 임의 추정 금지.

## 후속 요청

- "방금 그 섹션 다른 이름으로" — `--section` 만 바꿔서 다시 호출
- "다른 티켓에도 같은 내용" — 키만 바꿔서 같은 흐름 재실행
- "내용 살짝 수정해서" — 사용자 수정 받은 뒤 다시 미리보기 → Y/n

## 실패 처리

- `tako show` / `tako update` 가 비-제로 종료 → stderr 그대로 노출 후 중단
- 인증 실패(401/403) → `tako init` 으로 자격 증명 갱신 안내
- 본문 변환 실패 (드물지만 ADF 라운드트립에서 특수 노드 손실 가능) → 사용자에게 *변환 후 모습*을 한 번 더 보여주고 진행 여부 묻기

## 민감 정보 주의

업데이트 결과는 *Jira 본문*에 영구 기록된다. 세션 안의 토큰·비밀번호·내부 식별 정보가 *그대로 노출*될 수 있으므로 민감 영역 작업 후엔 미리보기 단계에서 신중히 검토.

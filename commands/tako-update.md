# /tako-update — 세션 맥락 → 티켓 제목/본문 반영

세션에서 한 작업을 *티켓 본문(description)* 또는 *제목(summary)* 에 기록한다. 부수 가치 아닌 *명세 누적*이라는 한 가지에 집중. 코멘트 추가가 아니라 *티켓 자체*를 건드리는 점에서 위험이 있으므로 미리보기 → Y/n 강제.

## 작동 환경 가정

- `tako init` 완료. 본인 사이트 인증 가능 상태
- 세션 컨텍스트가 *있어야* 의미 있음. 빈 세션이면 사용자에게 안내하고 종료
- 대상 프로젝트는 team-managed 가정

## 흐름

### 1. 인자 정규화

사용자 인자에서 키 추출 (WL-8876 / browse URL 둘 다 OK). 형태 안 맞으면 사용자에게 한 번 확인.

### 2. 현재 본문 조회 + 세션 맥락 요약

- `tako show <KEY> --json` 으로 현재 본문 받음 (모드 분기·미리보기에 활용)
- 동시에 세션 맥락에서 *진행 상황·결정 사항·변경 파일* 등을 한국어 마크다운으로 정리. *최상단 두 섹션은 반드시 포함*:
  ```markdown
  [내가 한 일]
  - <세션에서 실제로 한 작업을 *무엇을·왜* 중심으로 한 줄. 코드·파일명 나열 대신 기획자도 이해할 평이한 말로>
  - <검증 완료된 사실만>

  [현재 상태/결론]
  - <지금 상태: 완료 / 진행 중 / 막힘 / 후속 필요>
  - <남은 작업·다음 액션이 있으면 한 줄>

  (옵션) ## 영향 범위 — 단, 세션에서 *직접 확인한* 영향만
  ```
  - `[ ]` 라벨 형태 그대로 — 마크다운 헤더(`#`) 가 아님. ADF 변환 시 paragraph 로 나가지만 사람 눈엔 *섹션 마커* 로 보임.
  - append 모드는 `## 업데이트 (YYYY-MM-DD)` *섹션 안* 에 위 두 섹션을 둔다. overwrite 모드는 *통째 새 본문* 의 최상단에 두 섹션.
- 세션이 비었거나 너무 짧으면 *진행하지 말고* 사용자에게 묻는다.

### 2.1 작성 원칙 (반드시 지킬 것)

본문·제목 모두 *개발자가 아닌 사람(기획자 등)도 읽고 이해할 수 있어야 하는* 문서다. AI 가 만든 문장이라도 *사람이 직접 쓴 것처럼 자연스럽게* 읽혀야 하며, 검증되지 않은 내용으로 본문을 부풀리지 않는다.

**제목 (`--summary` 변경 시)** — 작업 *개요* 가 한 줄로 드러나게.

- 형태: *대상 + 동사(또는 결과)*. 명사구 60자 이내.
- 모호 단어 단독 금지: "기능 개선", "버그 수정", "리팩터링", "정리".
- 좋은 예: "보드 정렬 토큰 격리로 다른 컬럼 영향 제거" / "WL-9200 자식 이슈 일괄 조회 JQL 추가".
- 나쁜 예: "버그 수정" / "정렬 기능 개선".

**본문 (`--body` 변경 시)** — *세션에서 실제로 한 작업 / 검증된 사실* 만.

- 최상단에 두 섹션 필수: `[내가 한 일]` / `[현재 상태/결론]`. append 모드면 `## 업데이트 (YYYY-MM-DD)` 섹션 *안* 의 최상단, overwrite 모드면 새 본문 자체의 최상단.
- 세션 안에서 *실제로 실행했거나 코드에서 확인한 사실* 만 쓴다. 실행 안 한 가설·일반론·AI 추측은 빼거나 `> 미검증:` 같은 별도 라벨로 분리.
- 본 작업과 *직접 관련 없는 주변 분석* 은 빼기 (예: 변경한 함수와 무관한 코드 품질 평가, 보지도 않은 모듈 추측).
- "~일 것 같다", "추정", "아마도", "may", "could" 같은 표현이 들어가면 — *그 문장을 빼거나* 사용자에게 묻는다.
- 본문 길이는 *실제 작업 분량에 비례*. 작은 변경에 일반론 길게 늘이지 않는다.
- **깔끔한 마크다운으로.** 불릿·짧은 문단 위주로 읽기 좋게. (단 최상단 두 섹션 마커 `[ ]` 는 `#` 헤더가 아님 — 위 형식 그대로.)
- **코드·로그를 그대로 붙여넣지 않는다.** AI 코딩 도구 출력이든 직접 친 코드든 raw 코드·스택트레이스 통째 복붙 금지. 꼭 필요한 식별자(파일·함수명) 정도만 남기고 *무엇을·왜* 했는지 평이한 문장으로 푼다.
- **개발자가 아닌 기획자도 이해할 언어로.** 기술 디테일은 최소화하고 *의미*(무슨 문제를 어떻게 풀었는지) 중심으로 쓴다. AI 가 찍어낸 듯한 번역체·정형구 말고 *사람이 직접 쓴 것처럼* 자연스럽게. 단, 자연스럽게 쓴다고 위 분량 비례 원칙을 어기며 길게 늘이지 않는다.

### 3. 모드 결정

- 기본 **append** — 기존 본문 끝에 `## 업데이트 (YYYY-MM-DD)` 섹션 추가
- 사용자가 "통째 다시 써", "명세 재작성" 같은 표현 시에만 **overwrite** 분기 (`--mode overwrite`)
- 섹션 이름을 사용자가 지정하면 (`--section "진행 상황"` 등) 그대로 사용
- 사용자가 "제목도 바꿔줘", "제목만 ~로", "summary 를 ~로" 같이 *제목 변경* 의도 → `--summary "<새 제목>"` 인자 같이 전달. 본문 손대지 말라고 했으면 `--summary` 만, 같이 바꾸라고 했으면 `--summary` + `--body`

### 3.5 자가 점검 — 미리보기 *직전*

미리보기 띄우기 *전에* 새 제목·새 섹션·새 본문을 너 자신이 한 번 훑는다 (출력 없이). 다음이 *하나라도* 걸리면 미리보기에 *짧은 경고 라인* 을 같이 띄워 사용자가 결정하게 한다 — 자동 수정 금지.

체크리스트:

1. **제목이 작업 개요를 보여주는가?** (`--summary` 변경 시) 모호 단어("개선", "수정", "리팩터링", "정리") 단독이면 경고.
2. **제목이 너무 짧은가?** (`--summary` 변경 시) 10자 미만이면 경고 — 대개 정보 부족.
3. **새 본문/섹션 최상단에 필수 두 섹션이 있는가?** (`--body` 변경 시) `[내가 한 일]` 과 `[현재 상태/결론]` 둘 다 있어야 함. append 모드면 `## 업데이트 (YYYY-MM-DD)` 섹션 안의 최상단, overwrite 모드면 새 본문 자체의 최상단. 하나라도 누락 / 내용 한 줄 미만이면 경고.
4. **본문/섹션에 미검증 표현이 있는가?** "~일 것 같다", "추정", "아마도", "may", "could" 등. 발견 시 어느 줄인지 같이 알려주기.
5. **본문/섹션이 세션 작업 분량 대비 과한가?** 한 함수 수정인데 모듈 전반 분석이면 경고.
6. **본 작업과 무관한 분석이 섞였는가?** 사용자가 *명시적으로 요청* 하지 않은 주변 코드 평가·미실행 가설은 빼거나 묻기.
7. **append 새 섹션이 기존 본문과 중복되는 일반론인가?** 이미 본문에 적힌 배경 설명을 반복해 늘이지 말 것.
8. **코드 복붙·전문용어 과다인가?** raw 코드/로그가 통째로 들어갔거나, 기획자가 못 알아볼 전문용어 위주면 경고 — 평이한 언어로 풀지 사용자에게 묻기.

경고 출력 형태 (미리보기와 같이):

```
[자가 점검 — 사용자 결정 부탁]
  - 새 섹션 최상단에 [내가 한 일] / [현재 상태/결론] 두 섹션이 누락됐습니다. 추가하고 진행할까요?
  - 본문 "## 영향 평가" 부분은 세션에서 검증한 내용이 아니라 일반 분석입니다. 빼고 진행할까요?
  - 제목 "정렬 개선" 은 작업 결과가 잘 안 보입니다. 그대로 진행할지 알려주세요.
  - 새 섹션에 코드 블록이 통째로 들어가 있습니다. 기획자도 이해할 평이한 설명으로 바꿔 진행할까요?
```

원칙:
- *경고만*. 자동 수정·자동 삭제 금지.
- 걸리는 게 없으면 조용히 미리보기로 진행.

### 4. 미리보기 + 확인

`tako update <KEY> --body "..." --section "..."` 호출 *없이* 먼저 사용자에게 *추가될 섹션* 또는 *교체될 본문* 을 한국어로 보여주고 Y/n. 사용자가 동의하면 다음 단계로.

### 5. 적용 — `tako update` 호출

확인 받으면 인자 묶어서 호출. `--yes` 로 셸 자체 확인은 건너뛴다 (슬래시 커맨드가 이미 받았으므로 두 번 묻지 않음).

```bash
tako update <KEY> \
  --section "<섹션 이름>" \
  --mode append \
  --body "$(cat <<'BODY'
- 작업 내용 1
- 작업 내용 2
BODY
)" \
  --yes
```

overwrite 모드면 `--mode overwrite` + `--body` 에 *완성된 새 본문* 전체.

제목만 변경:
```bash
tako update <KEY> --summary "<새 제목>" --yes
```

제목 + 본문 동시 변경:
```bash
tako update <KEY> \
  --summary "<새 제목>" \
  --body "..." \
  --mode overwrite \
  --yes
```

`--summary` 와 `--body` 중 *최소 하나는* 있어야 함. `--mode` 는 본문에만 영향 — 제목은 항상 교체.

### 6. 결과 보고

```
업데이트 완료: <KEY>
  링크: https://<site>/browse/<KEY>
```

## 의사결정 사전 상의 (반드시)

다음 상황에서 LLM 이 추정하지 말고 사용자에게 묻는다:

- **세션 컨텍스트가 비었거나 매우 짧음** — "업데이트로 기록할 작업 흐름이 세션에 보이지 않습니다. 직접 본문을 알려주시거나 더 작업하신 뒤 다시 호출해 주세요."
- **세션이 여러 갈래** — 한 세션에 *여러 작업 흐름*이 섞여 있을 때. 어느 흐름을 본문에 반영할지 한 번 확인.
- **overwrite 모드 의도 불분명** — 사용자가 "업데이트해줘" 같은 모호한 표현. 기본은 append, 사용자가 *명시적으로 통째 재작성*을 원하는지 한 번 확인.
- **본문 현재 상태가 *명세성 정보로 가득* 한 경우** — overwrite 면 명세 손실. 사용자가 의도한 게 맞는지 추가 확인.
- **제목 변경 의도가 모호** — 사용자가 "제목 좀 바꿔" 정도로만 말하면 *현재 제목 그대로 보여주고* 새 제목을 받는다. 세션 맥락에서 임의 추정 금지.

## 후속 요청

- "방금 그 섹션 다른 이름으로" — `--section` 만 바꿔서 다시 호출
- "다른 티켓에도 같은 내용" — 키만 바꿔서 같은 흐름 재실행
- "내용 살짝 수정해서" — 사용자 수정 받은 뒤 다시 미리보기 → Y/n

## 실패 처리

- `tako show` / `tako update` 가 비-제로 종료 → stderr 그대로 노출 후 중단
- 인증 실패(401/403) → `tako init` 으로 자격 증명 갱신 안내
- 본문 변환 실패 (드물지만 ADF 라운드트립에서 특수 노드 손실 가능) → 사용자에게 *변환 후 모습*을 한 번 더 보여주고 진행 여부 묻기

## 민감 정보 주의

업데이트 결과는 *Jira 본문*에 영구 기록된다. 세션 안의 토큰·비밀번호·내부 식별 정보가 *그대로 노출*될 수 있으므로 민감 영역 작업 후엔 미리보기 단계에서 신중히 검토.
